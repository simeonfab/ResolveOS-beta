# ResolveOS Migration Strategy

## Core Principle

ResolveOS is extracted from existing project knowledge.

ResolveOS is not created from scratch.

Existing guidance must be preserved, classified, standardised, and only then refactored.

## Migration Principles

- Extract, don't invent.
- Preserve before simplifying.
- Do not flatten specialist roles.
- Preserve wording where wording matters.
- Use admin rather than personal names in global files.
- Work in reviewable batches.
- Do not skim.
- Project doctrine stays in project repositories.
- Global content must be genuinely reusable.

## Migration Order

### Batch 0 - Foundation

Create standards, metadata rules, context-loading rules, governance rules, and source-of-truth rules.

### Batch 1 - Admin Operating Rules

Create reusable operating rules, AI governance, communication rules, blocker handling, review discipline, and cost-safe AI guidance.

### Batch 2 - Core Skills

Create ticket-writing, acceptance criteria, feedback processing, completion reporting, and related reusable skills.

### Batch 3A - Standard Roles

Create lower-risk role files.

### Batch 3B - Specialist Roles

Create high-risk specialist roles using near 1:1 migration.

### Batch 4 - Templates

Create reusable templates.

### Batch 5 - ResolvePM Refactor

Remove duplication only after ResolveOS has become the source of truth.

## Promotion Rules

Content may move into ResolveOS only when:

- Reusable across multiple projects.
- Not tied to a single product.
- Not tied to a single roadmap.
- Not tied to a specific feature.
- Not tied to ResolvePM implementation decisions.

## Role Preservation Rules

Technical Strategy Lead remains Technical Strategy Lead.

Implementation Engineer remains Implementation Engineer.

Strategic Product Director remains Strategic Product Director.

No role consolidation without explicit review.

## Review Gates

Every migration batch requires review before the next batch begins.

No automatic bulk migration is permitted.
